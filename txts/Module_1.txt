1.1 What Problem Elasticsearch Solves


Search vs Database Mindset

Relational databases are optimized for transactional consistency (ACID), structured queries, and
row-based storage. They are not optimized for full-text search across large unstructured datasets.


•   RDBMS optimized for transactions

•   Row-based storage model

•   Indexes optimized for equality/range

•   LIKE queries cause scans at scale


Elasticsearch is designed specifically for search workloads where:


•   Text search is primary operation

•   Relevance ranking matters

•   Data volume is large

•   Horizontal scaling is required



Why LIKE Queries Fail at Scale

        SELECT * FROM products
        WHERE name LIKE '%shoe%'



This query cannot efficiently use traditional B-tree indexes due to leading wildcard. Database scans
rows. Elasticsearch instead uses inverted index lookup.


1.2 Lucene & Inverted Index Internals


What Lucene Actually Does
Lucene is the core indexing engine. Elasticsearch distributes Lucene across nodes.


•   Tokenization of text

•   Creation of inverted index

•   Segment storage

•   Relevance scoring



Inverted Index Deep Explanation

        Documents:
        Doc1: "red shoe"
        Doc2: "blue shoe"
        Doc3: "red hat"

        Inverted Index:

        red    -> Doc1, Doc3
        blue   -> Doc2
        shoe   -> Doc1, Doc2
        hat    -> Doc3



Search becomes set operations on posting lists. AND queries perform intersections.


Term Dictionary & Posting List

Term dictionary stores unique terms. Posting list stores document IDs for each term.


Tokenization & Analyzers


•   Tokenizer splits text into tokens

•   Analyzer may lowercase, stem, remove stopwords

•   Text fields are analyzed

•   Keyword fields are not analyzed



1.3 Core Data Model


Document, Field, Index, Mapping


•   Document = JSON object

•   Field = Key-value pair

•   Index = Logical grouping

•   Mapping defines data types
Text vs Keyword (Deep Difference)

        Field: name

        Type: text
        - Analyzed
        - Used for full-text search

        Type: keyword
        - Not analyzed
        - Used for exact match & aggregations



Use text for search, keyword for filtering and aggregations.


1.4 Distributed Architecture


Node Roles


•   Master – Manages cluster state

•   Data – Stores shards

•   Ingest – Preprocess data

•   Coordinating – Routes requests



Cluster State & Shard Allocation

        Cluster State:
        - Index metadata
        - Node list
        - Shard allocation map



Master node updates cluster state and allocates shards across nodes.


Primary vs Replica Mechanics

        Write Operation:

        Client -> Primary Shard
                     |
                     v
                 Replica Shard



Writes go to primary. Replicas sync after primary acknowledgement.
