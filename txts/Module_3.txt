Indexing is near real-time. Documents are written to an in-memory buffer and periodically flushed
into immutable segments.


Refresh Cycle


•   Default refresh interval: 1 second

•   Refresh makes new documents searchable

•   Refresh is lightweight (not full disk flush)



Segment Creation & Merge Process

           |

Segments are immutable. Updates create new documents and mark old ones deleted. Merge
process reduces segment count and improves search performance.
3.2 Bulk API – Throughput vs Latency

Bulk API batches multiple indexing operations in a single request.

        POST _bulk
        { index }
        { doc1 }
        { index }
        { doc2 }
        { index }
        { doc3 }



•   Higher throughput

•   Reduced HTTP overhead

•   Better resource utilization

•   Tradeoff: Slightly higher latency per document


In production systems, bulk indexing is mandatory for performance.


3.3 Logstash Architecture

        Input   --->    Filter    --->   Output

        Example:
        File Input ---> Grok Filter ---> Elasticsearch Output



•   Input: Reads data (file, kafka, jdbc)

•   Filter: Transforms data (grok, mutate, date)

•   Output: Sends to ES, Kafka, etc.


Logstash acts as ETL pipeline before data reaches Elasticsearch.

3.6 Mapping Strategy


Dynamic Mapping Risks


•   Automatic field creation

•   Unexpected data types

•   Mapping explosion



Mapping Explosion

Occurs when too many unique fields are created dynamically (e.g., JSON with variable keys).


Fielddata & Doc Values


•   Doc values used for sorting & aggregations

•   Fielddata loads text fields into memory

•   Improper usage can increase heap usage
