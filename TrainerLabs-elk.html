Lab 1: Exploring JSON and REST APIs
Verify Ubuntu env, practice curl, jq, and HTTP verbs
▶

Install tools: curl is the command-line HTTP client we use to talk to Elasticsearch. jq is a JSON formatter that makes API responses readable.

Terminal
sudo apt-get update && sudo apt-get install -y curl jq
▶ Expected Output
# Expected: Reading package lists... Done
# Installing curl jq...
# The following NEW packages will be installed:
#   curl jq

Verify curl is installed: This confirms curl is on your PATH and ready to use.

Terminal
curl --version
▶ Expected Output
curl 7.88.1 (x86_64-pc-linux-gnu)
Release-Date: 2023-02-20
Protocols: dict file ftp ftps http https ...

Create a JSON document file: We write a sample JSON object to a file so we can experiment with the jq tool and later with curl.

Terminal
echo '{"title": "Intro to ES", "author": "Trainer", "year": 2024}' > doc1.json
▶ Expected Output
# (no output — file created silently)

Pretty-print JSON with jq: The dot (.) tells jq to output the entire object, nicely formatted with colour and indentation.

Terminal
cat doc1.json | jq .
▶ Expected Output
{
  "title": "Intro to ES",
  "author": "Trainer",
  "year": 2024
}

Practice all 4 HTTP verbs: GET retrieves data, POST creates/submits, PUT replaces/updates, DELETE removes. httpbin.org is a free echo service — it simply mirrors back what you send.

Terminal
# GET
curl -s https://httpbin.org/get | jq .url
# POST
curl -s -X POST https://httpbin.org/post -H 'Content-Type: application/json' -d '{"name":"test"}' | jq .json
# PUT
curl -s -X PUT https://httpbin.org/put -H 'Content-Type: application/json' -d '{"name":"updated"}' | jq .json
# DELETE
curl -s -X DELETE https://httpbin.org/delete | jq .url
▶ Expected Output
"https://httpbin.org/get"
{"name": "test"}
{"name": "updated"}
"https://httpbin.org/delete"
💡 jq is a command-line JSON processor. jq . pretty-prints; jq .field extracts a field.

Lab 3: Installing Elasticsearch & Kibana on Ubuntu
APT install, systemd service, verify health endpoint
▶
⚠️ Write down the auto-generated elastic password shown during installation!

Install prerequisites: apt-transport-https allows APT to use HTTPS repositories. wget is used to download the GPG signing key in the next step.

Terminal
sudo apt-get update && sudo apt-get install apt-transport-https wget
▶ Expected Output
# Reading package lists... Done
# Building dependency tree... Done

Import Elastic's GPG signing key: This key proves that the packages you download are genuinely from Elastic and have not been tampered with. The --dearmor flag converts the ASCII key to binary format that APT expects.

Terminal
wget -qO - https://artifacts.elastic.co/GPG-KEY-elasticsearch | sudo gpg --dearmor -o /usr/share/keyrings/elasticsearch-keyring.gpg
▶ Expected Output
# (no output — key saved silently to /usr/share/keyrings/)

Add the Elastic 8.x APT repository: This tells APT where to find Elasticsearch and Kibana packages. The signed-by= constraint means APT will reject packages not signed by the key we just imported.

Terminal
echo "deb [signed-by=/usr/share/keyrings/elasticsearch-keyring.gpg] https://artifacts.elastic.co/packages/8.x/apt stable main" | sudo tee /etc/apt/sources.list.d/elastic-8.x.list
▶ Expected Output
deb [signed-by=/usr/share/keyrings/elasticsearch-keyring.gpg] https://artifacts.elastic.co/packages/8.x/apt stable main

Install Elasticsearch and Kibana: APT downloads and installs both packages. On first install, Elasticsearch auto-configures TLS and prints the elastic superuser password — copy it now!

Terminal
sudo apt-get update && sudo apt-get install elasticsearch kibana
▶ Expected Output
# Setting up elasticsearch (8.10.4) ...
# --------------------------- Security autoconfiguration information ------------------------------
# Password for the elastic user: <COPY THIS>
# Setting up kibana (8.10.4) ...

Enable and start the Elasticsearch service: daemon-reload makes systemd aware of the new unit file. enable makes it start automatically on boot. start launches it right now.

Terminal
sudo systemctl daemon-reload
sudo systemctl enable elasticsearch.service
sudo systemctl start elasticsearch.service
▶ Expected Output
# Synchronizing state of elasticsearch.service with SysV service script...
# Created symlink /etc/systemd/system/multi-user.target.wants/elasticsearch.service

Verify Elasticsearch is running: We use the auto-generated CA certificate to verify TLS. You will be prompted for the elastic password. A JSON response with cluster_name confirms success.

Terminal
curl --cacert /etc/elasticsearch/certs/http_ca.crt -u elastic https://localhost:9200

Lab 5: Simulating & Advanced Troubleshooting
Intentionally create yellow cluster, diagnose with allocation API
▶

Check the baseline cluster health: On a freshly installed single-node cluster this should return green (all primaries assigned, no replicas needed because number_of_replicas defaults to 0 for system indices).

Kibana Dev Tools
GET _cluster/health
▶ Expected Output
{
  "cluster_name" : "elasticsearch",
  "status" : "green",
  "number_of_nodes" : 1,
  "active_primary_shards" : 3
}

Intentionally force a YELLOW state: By requesting 1 replica on a single-node cluster, Elasticsearch cannot place the replica on a different node (same-shard constraint). The replica stays UNASSIGNED, turning the cluster yellow.

Kibana Dev Tools
PUT /troubleshoot_index
{
  "settings": { "number_of_replicas": 1 }
}
▶ Expected Output
{"acknowledged":true}

Observe the cluster go YELLOW: The status field will now show yellow. unassigned_shards will be 1 because the replica for troubleshoot_index shard 0 cannot be placed anywhere.

Kibana Dev Tools
GET _cluster/health
▶ Expected Output
{
  "status" : "yellow",
  "unassigned_shards" : 1,
  "active_primary_shards" : 4
}

Diagnose the unassigned shard using the Allocation Explain API: This is the most useful troubleshooting API in ES. It tells you exactly WHY a shard cannot be placed, e.g. 'the shard cannot be allocated to the same node as the primary'.

Kibana Dev Tools
GET _cluster/allocation/explain
▶ Expected Output
{
  "index" : "troubleshoot_index",
  "shard" : 0,
  "primary" : false,
  "current_state" : "unassigned",
  "unassigned_info" : { "reason" : "INDEX_CREATED" },
  "can_allocate" : "no",
  "allocate_explanation" : "cannot allocate because allocation is not permitted to any of the nodes"
}

Check node resource usage: Heap percent above 85% causes GC pressure. Disk above 85% triggers the low disk watermark and stops new shard allocation.

Kibana Dev Tools
GET _cat/nodes?v&h=name,heap.percent,ram.percent,cpu,disk.used_percent
▶ Expected Output
name         heap.percent ram.percent cpu disk.used_percent
your-node              42          58   3               12

Check thread pool queues: If queue > 0 or rejected > 0 for the write or search pools, your node is under load. Rejected = requests dropped — a critical sign that needs immediate attention.

Kibana Dev Tools
GET _cat/thread_pool/search,write?v&h=node_name,name,active,queue,rejected
▶ Expected Output
node_name    name    active queue rejected
your-node    search       0     0        0
your-node    write        0     0        0
Cluster Status:
  GREEN  — All primary AND replica shards assigned
  YELLOW — All primaries OK, some replicas unassigned
  RED    — One or more primary shards unassigned (DATA LOSS RISK!)

Execute the Bulk API request: --data-binary preserves the exact newlines in the file (unlike -d which strips them). The @ prefix tells curl to read from a file. You will be prompted for the elastic password.

Create the Logstash pipeline configuration file
Copy and paste the block below — it writes the config file directly, no editor needed. Replace <PASSWORD> with your elastic password.
• input: reads lines you type in the terminal
• filter: automatically adds a processed_by field to every event
• output: sends to Elasticsearch AND prints to screen so you can see results live

Terminal
cat <<'EOF' | sudo tee /etc/logstash/conf.d/my_pipeline.conf
input { stdin { } }
filter {
  mutate { add_field => { "processed_by" => "logstash_lab" } }
}
output {
  elasticsearch {
    hosts => ["https://localhost:9200"]
    user => "elastic"
    password => "<PASSWORD>"
    cacert => "/etc/elasticsearch/certs/http_ca.crt"
    index => "logstash_test_index"
  }
  stdout { codec => rubydebug }
}
EOF
▶ Expected Output
# /etc/logstash/conf.d/my_pipeline.conf written
# Verify: sudo cat /etc/logstash/conf.d/my_pipeline.conf

Run Logstash in the foreground and test it: Wait until you see 'Pipeline main started', then type a message and press Enter. You should immediately see the parsed JSON event printed to your terminal — with your message plus the processed_by field added by the filter.

Terminal

# Press Ctrl+C to stop

Lab 8: Installing and Configuring Logstash
CSV file -> Logstash pipeline -> Elasticsearch
▶

Install Logstash and verify the version: Logstash is the ETL pipeline of the Elastic Stack. It can ingest from hundreds of sources (files, databases, Kafka, syslog) and output to ES, S3, HDFS and more.

Terminal
sudo apt-get install -y logstash
sudo /usr/share/logstash/bin/logstash --version
▶ Expected Output
logstash 8.10.4

Create the Logstash CSV pipeline configuration
Copy and paste the block below — it writes the config file directly, no editor needed. Replace <PASSWORD> with your elastic password.
• input: reads the CSV file from disk, starting from the first line every time
• filter: splits each row into named fields, then converts salary to a number (so Elasticsearch stores it as an integer, not text)
• output: sends each record to Elasticsearch AND prints it to the screen

Terminal
cat <<'EOF' | sudo tee /etc/logstash/conf.d/csv-pipeline.conf
input {
  file {
    path => "/tmp/employees.csv"
    start_position => "beginning"
    sincedb_path => "/dev/null"
  }
}
filter {
  csv {
    separator => ","
    columns => ["employee_id","name","department","salary"]
  }
  mutate {
    convert => { "salary" => "integer" }
    remove_field => ["message","event","log","@version"]
  }
}
output {
  elasticsearch {
    hosts => ["https://localhost:9200"]
    user => "elastic"
    password => "<PASSWORD>"
    cacert => "/etc/elasticsearch/certs/http_ca.crt"
    index => "employees"
  }
  stdout { codec => rubydebug }
}
EOF
▶ Expected Output
# /etc/logstash/conf.d/csv-pipeline.conf written

Create the sample employee data file

Important: the column order here must exactly match the columns list in the pipeline config above.

Terminal

1,Alice Johnson,Engineering,95000
2,Bob Smith,Marketing,72000
3,Carol Williams,Engineering,105000

EOF
▶ Expected Output

# Verify: cat /tmp/employees.csv

▶

Insert sample data

Mixed context: must (scored) + filter (cached)

Kibana Dev Tools
GET products/_search
{
  "query": {
    "bool": {
      "must":   { "match": { "name": "shoe" } },
      "filter": { "range": { "price": { "lt": 100 } } }
    }
  }
}

All in filter: _score = 0.0 (faster, cached)

Kibana Dev Tools
GET products/_search
{
  "query": {
    "bool": {
      "filter": [
        { "match": { "name": "shoe" } },
        { "range": { "price": { "lt": 100 } } }
      ]
    }
  }
}
Bool Query Clauses:
  must      -> scored match  (affects _score)
  should    -> scored boost  (optional)
  filter    -> binary match  (fast, cached, no score)
  must_not  -> exclude docs  (no score impact)
